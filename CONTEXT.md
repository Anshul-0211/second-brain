# CONTEXT.md — Second Brain API & Frontend Specification

**Project:** Second Brain — AI-powered memory management system  
**Backend:** Node.js + TypeScript (Hono framework, Supabase PostgreSQL)  
**Current Frontend:** Flutter mobile app  
**Target:** Next.js fullstack implementation  
**Date Generated:** April 2026

---

## 1. API Reference

### Global Configuration
- **Base URL:** `http://localhost:3000` (local dev) → Production Render deployment
- **Server Port:** `3000` (or `PORT` env variable)
- **Framework:** Hono v4.7.0 (lightweight, modular routing)

### CORS & Middleware
- **CORS:** Allows all origins (`*`) with methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
- **Allowed Headers:** `Content-Type`, `x-api-key`
- **Global Middleware:**
  - `logger()` — HTTP request logging
  - `cors()` — Cross-origin resource sharing
  - Global 404 handler → JSON `{ error: 'Not Found' }` (404)
  - Global error handler → JSON `{ error: 'Internal Server Error' }` (500)

### Error Response Format
All errors return JSON with status code:
```json
{
  "error": "<error message>"
}
```
Possible status codes: **400** (bad request), **401** (unauthorized), **404** (not found), **500** (server error)

#### Search

##### Semantic/Hybrid Search
```
POST /api/search
```
**Body:**
```json
{
  "query": "string (required)",
  "threshold": 0.4 (optional, 0.0-1.0 similarity score),
  "limit": 10 (optional, max results)
}
```
**Response:** `200 OK`
```json
{
  "query": "javascript performance",
  "searchMethod": "semantic|tag-based|tag-based (fallback)",
  "results": [
    {
      "id": "uuid",
      "type": "note",
      "content_raw": "string",
      "title": "string",
      "description": "string",
      "source_url": "string",
      "tags": [{ "id": "uuid", "name": "javascript" }],
      "categories": [{ "id": "uuid", "name": "Tech", "color": "#6366F1" }],
      "similarity": 0.92
    }
  ],
  "count": 5
}
```
**Search Strategy:**
1. First tries **semantic search** using vector embeddings (ML-powered)
2. Falls back to **tag-based search** if embeddings unavailable
3. Uses Cloudflare Workers AI for embeddings, Groq LLM as fallback
4. Indexes content by: title (ilike), description (ilike), content_raw (ilike), tags, categories

**Errors:**
- `400 Bad Request` — `{ "error": "Query is required" }`

## 2. Auth & Roles

### Authentication Strategy
**API Key Authentication** (simple, suitable for personal/single-user app)

### Token/Key Location
- **Header:** `x-api-key: <api_key>`
- **Query Parameter:** `?key=<api_key>` (alternative)

### Key Configuration
- **Environment Variable:** `API_KEY` (default: `"dev-key"`)
- **Validation:** Middleware `apiKeyAuth()` in [src/middleware/apiKey.ts](src/middleware/apiKey.ts)
- **Logic:** If header/query key ≠ configured `API_KEY`, respond with `401 Unauthorized`

### Authorization
**No role-based access control (RBAC).** All authenticated users have equal access to all protected endpoints.

### Public vs. Protected Routes
**Public (no auth required):**
- `GET /` — Health check
- `GET /health` — Health check

**Protected (API key required):**
- All `/api/*` routes (items, search, reminders, recommendations, tags, categories, profile)

### Token Expiry & Refresh
**No token expiry.** API key is static and does not expire. No refresh mechanism needed.

## 3. Data Models

### Item
Represents a single piece of *content* (note, link, or file).

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated from `gen_random_uuid()` |
| `type` | enum | ✓ | One of: `'link'`, `'note'`, `'file'` |
| `content_raw` | text | ✓ | Raw input content (1-50000 chars) |
| `title` | text | - | Auto-extracted or defaults to first line of content |
| `description` | text | - | AI-generated summary or user-provided |
| `source_url` | text | - | If item is a link or file, the URL |
| `ai_summary` | text | - | Generated by LLM during pipeline |
| `confidence_score` | float | - | 0.0-1.0, reflects classification confidence |
| `embedding` | vector(384) | - | 384-dimensional vector from sentence transformer |
| `created_at` | timestamp | ✓ | UTC, auto-set on insert |
| `updated_at` | timestamp | ✓ | UTC, auto-updated on modification |
| `opened` | boolean | - | Tracking: has user viewed this item? |
| `opened_at` | timestamp | - | Tracking: when was it first opened? |
| `view_count` | integer | - | Tracking: how many times viewed? |
| `last_viewed_at` | timestamp | - | Tracking: when was last view? |
| `files` | JSONB array | - | Array of file attachment objects |
| `file_count` | integer | - | Count of attached files |
| `has_attachment` | boolean | - | Flag for quick filtering |

### ItemNote
A contextual note attached to an item (separate from the item's content).

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated |
| `item_id` | UUID | ✓ | FK to `items(id)`, cascade delete |
| `content` | text | ✓ | Note text (1-5000 chars) |
| `urgency` | enum | ✓ | One of: `'urgent'`, `'important'`, `'low-priority'` |
| `created_at` | timestamp | ✓ | UTC, auto-set |
| `updated_at` | timestamp | ✓ | UTC, auto-updated |

### Entity
Automatically extracted named entities (tasks, deadlines, people, etc.).

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated |
| `item_id` | UUID | ✓ | FK to `items(id)`, cascade delete |
| `text` | text | ✓ | Original extracted text (e.g., "assignment", "Sarah", "Thursday") |
| `type` | text | ✓ | One of: `'TASK'`, `'DEADLINE'`, `'PERSON'`, `'PROJECT'`, `'PRIORITY'` |
| `value` | text | ✓ | Normalized version (e.g., "assignment", ISO date) |
| `confidence` | float | - | 0.0-1.0, extraction confidence (default: 0.8) |
| `metadata` | JSONB | - | Additional context (e.g., `{ "date": "2026-04-15", "priority": "high" }`) |
| `created_at` | timestamp | ✓ | UTC |

### Reminder
Task reminder with due date and status tracking.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated |
| `item_id` | UUID | ✓ | FK to `items(id)`, cascade delete |
| `task_name` | text | ✓ | Human-readable task description |
| `due_date` | timestamp | ✓ | ISO8601 deadline |
| `priority` | text | ✓ | One of: `'low'`, `'medium'`, `'high'`, `'urgent'` |
| `status` | text | ✓ | One of: `'pending'`, `'completed'`, `'snoozed'`, `'dismissed'` |
| `notification_sent` | boolean | - | Has notification been dispatched? (default: false) |
| `notification_sent_at` | timestamp | - | When was notification sent? |
| `reminder_triggered_at` | timestamp | - | When was reminder triggered by scheduler? |
| `completed_at` | timestamp | - | When marked complete? |
| `snoozed_until` | timestamp | - | When should snoozed reminder reappear? |
| `created_at` | timestamp | ✓ | UTC |
| `updated_at` | timestamp | ✓ | UTC |

### DailyRecommendation
Recommended item generated daily at 8am.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated |
| `item_id` | UUID | ✓ | FK to `items(id)`, cascade delete |
| `score` | float | ✓ | 0-100, recommendation relevance score |
| `reason` | text | ✓ | Why recommended (e.g., "`has_deadline`", "`unviewed`", "`related_to_recent`") |
| `metadata` | JSONB | - | Breakdown: `{ urgency_score: 80, freshness_score: 90, relevance_score: 70 }` |
| `dismissed_at` | timestamp | - | When user dismissed this recommendation |
| `created_at` | timestamp | ✓ | UTC, auto-set on generation |
| `expires_at` | timestamp | ✓ | UTC, 1 day after creation |

### RecommendationDismissal
Track user dismissals of recommendations.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID | ✓ | Auto-generated |
| `recommendation_id` | UUID | ✓ | FK to `daily_recommendations(id)` |
| `item_id` | UUID | ✓ | FK to `items(id)` |
| `reason` | text | - | Why dismissed (e.g., "not_relevant", "already_done", "not_interested", "read_later") |
| `dismissed_at` | timestamp | ✓ | UTC |

### Tag

| Field | Type | Required | Notes |

### ItemTag (Many-to-Many)
Junction table linking items to tags.

| Field | Type | Notes |

## 4. Pages & User Flows

### Page/Route Map

#### Core Routes
```
/                          — Feed/Dashboard (loads items + recommendations)
/dump                      — Quick capture/input screen
/feed                      — Feed view (all items with categories filter)
/search                    — Search interface (semantic + tag-based)
/reminders                 — Reminder list with urgency filters
/detail/:itemId           — Item detail view (full content + notes + entities + reminders)
/profile                   — User profile (sign-up/display name)
```

#### Derived Routes (from code analysis)
- `/` — Home/default route → redirects to `/feed`
- `/login` — Not present (API key only, no auth UI in current codebase)

### User Flows

#### Flow 1: Create & Dump Content
**Trigger:** User taps "Dump" tab  
**Steps:**
1. User enters content in text field
2. System auto-detects type: `link`, `note`, or `file`
3. User optionally attaches files (max 4 files, 50MB each)
4. User can copy from clipboard or pick files
5. User clicks "Save"
6. **API Call:** `POST /api/items` → System processes via pipeline
   - Extracts title, summary, tags, categories
   - Creates embedding for search
   - Extracts entities (tasks, deadlines)
   - Auto-creates reminders if deadlines detected
7. Show success toast + reload feed
8. **Result:** Item appears in feed, reminders show in reminder tab

**API Calls:**
- `POST /api/items` — Create item with pipeline processing
- `GET /api/reminders` — Load reminder count (on DumpScreen mount)

#### Flow 2: Browse Feed
**Trigger:** User taps "Feed" tab  
**Steps:**
1. Load all items paginated (limit=50, offset=0)
2. Display items as card list
3. Show category filter chips (All, Tech, Finance, etc.)
4. Filter items by category on selection
5. Load today's recommendations (separate section)
6. User taps item card
7. **Navigate:** → Item detail page

**API Calls:**
- `GET /api/items?limit=50&offset=0` — Fetch items on mount
- `GET /api/recommendations/today` — Fetch daily recommendations
- `POST /api/recommendations/:id/dismiss` — When user dismisses recommendation
- `PUT /api/items/:id/view` — Mark item as viewed when user navigates to detail

#### Flow 3: Search & Discover
**Trigger:** User taps "Search" tab  
**Steps:**
1. User enters search query in search bar
2. Debounce 500ms on input change
3. User submits (or reaches minimum characters)
4. **API Call:** `POST /api/search` → Returns semantic + tag-based results
5. Display results as list
6. Show search method used (semantic vs. tag-based fallback)
7. User taps result item
8. **Navigate:** → Item detail page

**API Calls:**
- `POST /api/search` — Semantic/hybrid search

#### Flow 4: View Item Detail
**Trigger:** User taps item from feed or search results  
**Steps:**
1. **Navigate:** → `/detail/:itemId`
2. Load enriched item (tags, categories, notes, entities, reminders)
3. **API Call:** `PUT /api/items/:id/view` — Mark as viewed
4. Display:
   - Full content
   - Extracted title, summary, category chips, tag chips
   - List of notes (with urgency indicators)
   - List of entities (tasks, deadlines, people)
   - List of reminders (filters by pending status)
   - File attachments (links to download)
5. User can add note: `POST /api/items/:id/notes`
6. User can edit note: `PUT /api/items/:id/notes/:noteId`
7. User can delete note: `DELETE /api/items/:id/notes/:noteId`
8. User can delete item: `DELETE /api/items/:id`
9. User can open links (if item is link type)

**API Calls:**
- `GET /api/items/:id` — Fetch enriched item
- `PUT /api/items/:id/view` — Mark as viewed
- `POST /api/items/:id/notes` — Add note
- `PUT /api/items/:id/notes/:noteId` — Update note
- `DELETE /api/items/:id/notes/:noteId` — Delete note
- `GET /api/items/:id/entities` — Fetch extracted entities
- `GET /api/items/:id/reminders` — Fetch item-specific reminders (if any)
- `DELETE /api/items/:id` — Delete item

#### Flow 5: Manage Reminders
**Trigger:** User taps "Reminders" tab  
**Steps:**
1. Load all pending reminders
2. Calculate urgency: overdue, today, tomorrow, upcoming
3. Display with priority colors/icons (urgent=🔴, high=🟠, medium=🟡, low=⚪)
4. User filters by urgency (all/overdue/today/upcoming)
5. User taps reminder
6. Options: Mark complete, snooze, dismiss
7. **API Call:** `PUT /api/reminders/:id` with status update
8. Reload reminder list

**API Calls:**
- `GET /api/reminders?limit=100` — Fetch pending reminders on mount
- `PUT /api/reminders/:id` — Update reminder status (pending/completed/snoozed/dismissed)
- `DELETE /api/reminders/:id` — Dismiss reminder (alternative to PUT with status=dismissed)

#### Flow 6: User Profile Setup (if needed)
**Trigger:** First app launch or profile screen  
**Steps:**
1. **API Call:** `GET /api/profile` — Check if profile exists
2. If exists: Display display_name
3. If not: Show sign-up prompt
4. User enters display name
5. **API Call:** `POST /api/profile` → Create profile
6. Show success, continue to app

**API Calls:**
- `GET /api/profile` — Fetch existing profile
- `POST /api/profile` — Create new profile

#### Flow 7: Daily Re-engagement (Automatic)
**Trigger:** Server-side scheduler (8am UTC daily)  
**Steps:**
1. Backend generates recommendations for the day
2. Scores items based on: urgency (deadlines), freshness (unviewed), relevance (similar tags)
3. Stores in `daily_recommendations` table with expiry (1 day)
4. Frontend fetches via `GET /api/recommendations/today`
5. Displays in feed or dedicated recommendations section
6. User can dismiss with optional reason

**API Calls:**
- `GET /api/recommendations/today` — Fetch today's recommendations
- `POST /api/recommendations/:id/dismiss` — Dismiss recommendation

## 5. Global State & Shared Data

### State to Persist Across Screens

#### Authentication State
- **What:** API key (for local storage in dev)
- **Where Used:** All API calls (every route)
- **How Fetched:** From environment variable or local storage
- **Persistence:** localStorage (key: `api_key`)

#### User Profile
- **What:** `{ id, display_name, created_at }`
- **Where Used:** Profile screen, possibly header display
- **How Fetched:** `GET /api/profile` on app init
- **Persistence:** Memory + optional localStorage
- **Caching:** Fetch once on init, refresh on profile screen mount

#### Items List (Feed)
- **What:** Array of enriched items (paginated)
- **Where Used:** Feed screen, search results comparison, item count
- **How Fetched:** `GET /api/items?limit=50&offset=0` on feed mount
- **Persistence:** Memory (reload on pull-to-refresh or manual refresh)
- **Caching:** 5-minute cache recommended; invalidate on create/update/delete

#### Categories (Predefined)
- **What:** Array of 8 built-in categories with colors
- **Where Used:** Feed category filter, item detail category chips, search results
- **How Fetched:** `GET /api/categories` once on app init
- **Persistence:** localStorage (unlikely to change)
- **Caching:** Permanent cache (fetch on app init, reuse)

#### Tags (Dynamic)
- **What:** Array of auto-generated tags
- **Where Used:** Item detail tag chips, tag-based search
- **How Fetched:** `GET /api/tags` on app init or on demand
- **Persistence:** localStorage (update frequency: low)
- **Caching:** Cache with TTL 1 hour, refresh on create item

#### Pending Reminders
- **What:** Array of reminders with status=pending
- **Where Used:** Reminder tab (main list), feed notification badge, detail screen
- **How Fetched:** `GET /api/reminders?limit=100` on reminders tab mount, on feed init
- **Persistence:** Memory (reload on update/dismiss)
- **Caching:** 1-minute cache; invalidate on status change

#### Today's Recommendations
- **What:** Array of daily recommendations (expires after 24h)
- **Where Used:** Feed view (separate section or carousel)
- **How Fetched:** `GET /api/recommendations/today` on feed mount daily
- **Persistence:** Memory
- **Caching:** Single daily cache (fetch once per 24h UTC period)

#### Item Detail (Selected Item)
- **What:** Single enriched item with full metadata
- **Where Used:** Detail screen only
- **How Fetched:** `GET /api/items/:id` on detail screen mount
- **Persistence:** Memory (per navigation)
- **Caching:** None (fresh fetch per navigation to avoid stale entity/reminder data)

#### Search Results (Temporary)
- **What:** Array of search results
- **Where Used:** Search screen only
- **How Fetched:** `POST /api/search` on query submit
- **Persistence:** Memory (cleared on clear search or tab change)
- **Caching:** None (results specific to query)

### State Management Architecture

**Recommended: React Context + Local Storage**

```
AppContext
├── auth: { apiKey: string }
├── user: { id: string; displayName: string } | null
├── items: Item[]
├── categories: Category[]
├── tags: Tag[]
├── reminders: Reminder[]
├── recommendations: DailyRecommendation[]
├── selectedItem: Item | null
├── searchResults: SearchResult[]
└── loading: { [key: string]: boolean }
```

**Fetch Strategy:**
- **On App Init:**
  1. Retrieve API key from localStorage or env
  2. Fetch user profile
  3. Fetch categories (cache in localStorage)
  4. Load items
  5. Load reminders (badge on reminder tab)

- **On Feed Mount:**
  1. Check cache before fetching items
  2. Fetch recommendations

- **On Detail Screen Mount:**
  1. Fetch enriched item (fresh, no cache)
  2. Mark as viewed

- **On Reminder Status Change:**
  1. Optimistically update local state
  2. Send API call
  3. Refetch pending reminders list

```json
        "storage_path": "items-attachments/...",
        "uploaded_at": "2026-04-13T10:30:00Z"
      }
    ],
    "file_count": 1,
    "has_attachment": true
  }
}
```

### File Storage Configuration

#### Bucket
- **Name:** `items-attachments` (in Supabase Storage)
- **Visibility:** Private (requires signed URLs)
- **Max File Size:** 50MB per file (enforced client-side + server-side)
- **Max Files Per Item:** 4

#### File Metadata (JSONB in Item)
```json
{
  "id": "94c3dc6e-34a9-4fa3-9a7b-f52e8c1fdbcf",
  "name": "research-paper.pdf",
  "url": "https://project.supabase.co/storage/v1/object/sign/items-attachments/...",
  "size": 2097152,
  "mime_type": "application/pdf",
  "storage_path": "items-attachments/94c3dc6e.pdf",
  "uploaded_at": "2026-04-13T10:30:00Z"
}
```

#### Supported MIME Types
- **Documents:** `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- **Spreadsheets:** `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
- **Images:** `image/png`, `image/jpeg`, `image/webp`, `image/gif`
- **Text:** `text/plain`, `text/markdown`, `text/csv`
- **Archives:** `application/zip`, `application/x-rar-compressed`

#### URL Generation
- **Format:** Supabase signed URL (time-limited, default 7 days)
- **Example:** `https://project.supabase.co/storage/v1/object/sign/items-attachments/path-to-file?token=...`
- **Expiry:** 7 days (auto-delete after expiry if not renewed)

#### Frontend Upload Handling
1. Gather files from file picker / clipboard
2. Validate: max 4 files, max 50MB each file
3. Send via `POST /api/items/upload` with multipart form
4. Receive signed URLs in response
5. Display in item as links
6. User can click to download (signed URL handles auth)

#### Download/Access
- Users click file link → Signed URL → Download directly from Supabase Storage
- No frontend caching of file; always fresh from storage

## 10. Pagination, Filtering & Sorting

### Pagination Convention

#### Query Parameters
| Parameter | Type | Default | Max | Notes |
|-----------|------|---------|-----|-------|
| `limit` | integer | 50 | 100 | Items per page |
| `offset` | integer | 0 | ∞ | Pagination offset (not cursor) |

#### Example Requests
```
GET /api/items?limit=50&offset=0          # Page 1
GET /api/items?limit=50&offset=50         # Page 2
GET /api/items?limit=50&offset=100        # Page 3
```

#### Response Envelope
```json
{
  "items": [ ... ],
  "count": 50,
  "limit": 50,
  "offset": 0
}
```
- `count` — Number of items in this page (may be < limit on last page)
- `limit` — Requested limit
- `offset` — Requested offset

### Search Pagination
```
POST /api/search
```
**Body:**
```json
{
  "query": "javascript",
  "threshold": 0.4,
  "limit": 10
}
```
**Response:**
```json
{
  "query": "javascript",
  "searchMethod": "semantic|tag-based",
  "results": [ ... ],
  "count": 10
}
```
**Note:** Search uses `limit` only (not offset). Implement infinite scroll or lazy-load if needed.

### Reminder Pagination
```
GET /api/reminders?limit=100&status=pending
```
**Response:**
```json
{
  "success": true,
  "reminders": [ ... ],
  "count": 10,
  "stats": { "overdue": 1, "today": 2, ... }
}
```

### Filtering Conventions

#### By Status (Reminders)
```
GET /api/reminders?status=pending
```
**Allowed Values:** `pending`, `completed`, `snoozed`, `dismissed`

#### By Category (Frontend-side filtering on items array)
No server-side filtering; frontend filters loaded items:
```js
items.filter(item => item.categories.some(c => c.name === 'Tech'))
```

#### By Urgency (Frontend-side filtering)
Filter reminders/notes by urgency field:
- Notes: `urgent`, `important`, `low-priority`
- Reminders: Priority + calculated urgency (`today`, `tomorrow`, `soon`, `normal`, etc.)

### Sorting Conventions

#### Items (Fixed Order)
- **Default:** `created_at DESC` (newest first)
- **No custom sort parameter** (frontend can re-sort client-side if needed)

#### Reminders (Fixed Order)
- **Default:** `due_date ASC` (soonest due first)

#### Entities (Fixed Order)
- **Default:** `created_at DESC` (newest extracted first)

#### Tags (Fixed Order)
- **Default:** `name ASC` (alphabetical)

#### Categories (Fixed Order)
- **Default:** `name ASC` (fixed built-in order)

### Total Count
- **Not returned in paginated endpoints** (count only in response)
- **Workaround for mobile:** Load all items with large limit to get total
- **Better approach:** Maintain item count in `GET /api/items` response (current implementation provides count only for items loaded, not total DB count)
